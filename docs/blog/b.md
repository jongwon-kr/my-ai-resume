## 들어가며

기획에 이어 본격적으로 Cursor와 함께 개발을 시작했다.
Agent, Skill
화면을 만들고 기능을 붙이는 과정이 너무 재밌어서 밤낮없이 달렸다.
정신을 차려 보니 하루 만에 Cursor 사용량 한도를 전부 썼다.
개발 도중에 갑자기 손발이 묶인 기분이었다.
지금은 구글 Gemini의 도움을 받아 개발과 기록을 이어가고 있다.
우여곡절이 많았지만 기획했던 MVP 구현을 끝내고 실전용 기능까지 완성했다.

## 내용

기획 명세를 하나씩 코드로 바꾸며 서비스를 구체화했다.
한 방에 다 짜기보다 단계별로 나누어 AI에게 지시했다.

**초기 뼈대 구축**

- Next.js, Supabase, shadcn/ui 조합을 골랐다.
- 회원가입, 구글 로그인, 데이터베이스 설계까지 백엔드 구축을 Supabase로 해결했다.
- 덕분에 인프라 구축 고민을 덜고 화면과 AI 연동에만 몰입할 수 있었다.

**핵심 이력서 빌더와 챗봇 확장**

- 처음 기획한 4개의 섹션을 경력, 학력, 자격증 등 9개 섹션으로 늘렸다. 작성 중 날아가지 않게 자동 저장도 붙였다.
- 원하는 대로 섹션의 위치를 바꾸거나 표시 여부를 커스터마이징 할 수 있게 했다.
- 챗봇에는 답변이 끊기지 않는 스트리밍을 넣고, 자주 묻는 질문은 AI가 의미를 매칭해서 정확히 답하게 만들었다.

**PDF 입출력과 완성도 카드**

- PDF 가져오기: 기존 PDF 이력서를 업로드하면 Gemini가 내용을 추출해서 자동으로 채워준다.
- PDF 다운로드: 완성한 웹 이력서를 PDF 양식으로 뽑아낼 수 있게 구현했다.
- 이력서 완성도 현황을 한눈에 파악할 수 있도록 대시보드에 배치했다.

**한국형 디테일과 가상 데이터 고정**

- 민감한 개인정보를 숨길 수 있는 토글 버튼을 추가했다.
- 챗봇이 모르는 질문은 작성자에게 메일처럼 보낼 수 있는 직접 문의하기 폼을 만들었다.
- 랜딩 페이지에는 실제 데이터 대신 가상의 개발자 프로필(@kimdev)을 추가해 두었다.

**외부 연동 제거와 프록시 전환**

- RSS나 깃허브 외부 소스를 긁어오는 방식은 유연성이 떨어져 과감히 지웠다. 대신 이름과 URL을 직접 넣는 자유 링크 형태로 바꿨다.
- 미들웨어와 서버 코드에서 헤더와 주소(slug)를 직접 제어하도록 구현해 보안과 유연성을 높였다.

**멀티 장비 세션 연속성과 CI/CD 파이프라인**

- 효율적인 개발을 위해 `.cursorrules` 파일을 활용했다. 작업 후 개발 일지와 현황 문서를 작성하는 지침을 추가하여 여러 환경에서 개발해도 맥락이 끊기지 않게 만들었다.
- 코드 품질 유지를 위해 GitHub Actions 기반 CI 파이프라인을 구축하고, 빌드와 테스트 자동 검증을 적용했다.
- 로그인부터 채팅까지 전체 흐름을 Playwright 기반 E2E 통합 테스트로 검증하는 체계를 마련했다.

## 배운 점

- 도구 의존도가 높으면 한계에 부딪혔을 때 타격이 크다. 강력한 AI 비서라도 물리적 쿼터가 있으니 항상 대안을 생각해야 한다.
- 프로젝트 상황을 명확하게 문서화해 두는 버릇이 빛을 발했다. 개발 일지와 현황 명세를 잘 적어 두니, Cursor에서 Gemini로 대화 상대를 바꿔도 맥락 손실 없이 바로 동기화가 가능했다.
- 기능을 무조건 늘리기보다 복잡한 외부 연동을 쳐내고 단순한 링크 입력으로 바꾼 결정이 제품의 안정성과 개발 속도 면에서 훨씬 이득이었다.
