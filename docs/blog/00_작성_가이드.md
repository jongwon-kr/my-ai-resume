# CloneCV 개발 과정 기록 가이드

버전: v1.4 | 작성일: 2026-07-09

CloneCV 개발 과정을 **누구나 가볍게 읽을 수 있는** 글로 남기기 위한 가이드입니다.  
초안은 `docs/blog/`에 두고, Tistory·Velog 등 외부 발행은 **수동**으로 합니다.

에이전트: `.cursor/skills/blog-writer/SKILL.md`

**독자 = 블로그 방문자.** repo를 아는 사람(나)용 메모가 아니다.

---

## 1. 파일 규칙

| 항목 | 규칙                                              |
| ---- | ------------------------------------------------- |
| 위치 | `docs/blog/`                                      |
| 이름 | `YYYY-MM-DD_주제.md` 또는 `NN_초안_주제.md`       |
| 언어 | 한국어                                            |
| 길이 | 본문 800~1500자 권장 (길면 배운 점만 남기고 축약) |
| 톤   | 가볍고, 재미 조금, 유익하게                       |

가이드 파일(`00_작성_가이드.md`)은 초안이 아닙니다.

---

## 2. 글 구조 (필수) — 쓰기 전 개요

섹션 제목은 **한글**로 쓴다. (HOOK/STORY 등 영문 헤딩 쓰지 말 것)
**모든 편 동일** — 편마다 헤딩 이름을 바꾸지 않는다.

1. **들어가며** — 이 글이 왜 나왔는지 (짧게). 서비스·주제가 있으면 한 줄로.
2. **내용** — 이번 편의 본문 (무엇을 / 왜 / 어떻게). 사실만, 쉽게.
3. **배운 점** — 교훈 3개 이내
4. **다음에** — 다음 할 일 + **다음 글이 궁금하게**

이미지는 발행할 때 플랫폼에 직접 업로드한다. **본문에 repo 경로를 적지 않는다.**

---

## 3. 글 잘 쓰는 법

1. **개요 먼저** — What & How. 정보 글은 번호·목차, 주장은 [주장+근거].
2. **재료가 전부** — 직접 경험, 확인된 정보, 생각한 것만. 없으면 쓰지 않는다.
3. **문장은 짧고 단순하게** — 접속사 줄이기, 같은 말 반복 금지, 핵심은 단문.
4. **신박함 한 스푼** — 뻔한 문장 대신 에피소드나 색다른 표현 하나.
5. **관심 있는 이야기만** — 관심 없는 주제는 억지로 쓰지 않는다.
6. **조급하지 않게** — 주제 밖으로 새면 자른다. 급하게 쓰면 고치는 시간이 더 든다.
7. **가운데점(·) 자제** — 쉼표, “이랑”, “그리고”로. 사람 말투에 가깝게.

개발 블로그: **경험(재료) + 짧은 문장 + 개요(들어가며/내용/배운 점/다음에)** 조합이 핵심.

---

## 4. 금지

- 회사명·직장·개인 식별 정보·사적 환경
- 과장된 성과·미구현 기능을 완료처럼 서술
- 제품 코드 대규모 수정 (블로그 에이전트는 글만)
- **repo 내부 전용 정보**를 본문에 쓰기
  - 예: `docs/01`, `.cursor/skills`, 커밋 해시, `@문서` 붙이기, 나만 아는 파일명
- 독자에게 “우리 저장소 인덱스”처럼 읽히는 문장
- 들어가며와 내용에서 같은 말 두 번 하기
- 바이브코딩, 에이전트, 벡터 DB 등 은어는 풀어서 쓰거나 빼기

---

## 5. 재료 (에이전트·작성자만 참고)

우선순위 (글에는 경로를 노출하지 않음):

1. 개발일지 최근 항목
2. git log / 관련 커밋 (사실 확인용)
3. 현황·기능 명세 (사실 확인용)
4. README·스크린샷 (발행 시 이미지로만)

일상에서 떠오른 문장·에피소드는 메모해 두고 글감으로 쓴다.

---

## 6. 호출 예시

```
이번 주 작업을 블로그 초안으로 작성해줘.
주제: [한 줄]
독자용으로만 — repo 경로·커밋 해시 넣지 마.
문장은 짧게, 가운데점(·) 쓰지 마.
docs/blog/ 아래에 저장.
```

---

## 7. 초안 체크리스트

- [ ] 들어가며 → 내용 → 배운 점 → 다음에 (전 편 동일 한글 헤딩)
- [ ] 사적·식별 정보 없음
- [ ] 구현된 내용·확인된 사실만
- [ ] 본문에 docs/·.cursor/·커밋 해시·내부 파일명 없음
- [ ] 문장 짧음, 반복·접속사 과다 없음, **· 남용 없음**
- [ ] 처음 보는 사람도 서비스·도구 이름이 이해됨
- [ ] 신박한 한 줄 또는 에피소드 있음
- [ ] 「다음에」가 다음 편을 기대하게 함
- [ ] 모르는 사람도 맥락 파악 가능
